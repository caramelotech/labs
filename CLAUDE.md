## Overview

Caramelo Labs is the **hub repository** for all Caramelo Tech labs. It holds the Astro + Starlight site structure and, at build time, aggregates notes from separate content repositories (ai-labs, java-labs, etc.). It is published to GitHub Pages at `https://caramelotech.com.br/labs`.

Content repositories contain only plain Markdown under `notes/` - no frontmatter, no dependencies. This repo fetches them and injects the frontmatter Starlight requires.

## Commands

```bash
npm install          # install dependencies
npm run fetch        # clone content repos from GitHub and assemble src/content/docs/
npm run fetch:local  # same, but copy from sibling local clones (../<slug>)
npm run dev          # dev server at localhost:4321 (run a fetch first)
npm run build        # production build (outputs to dist/)
npm run preview      # preview the production build locally
```

There are no tests or linting.

## Architecture

```
labs.config.json           # registry of content repos (slug, label, repo, branch, notesDir)
scripts/fetch-content.mjs  # clones/copies notes, injects frontmatter, generates sidebar
labs-sidebar.generated.json # sidebar built by the fetch script (gitignored)
src/content/docs/          # only index.mdx and 404.md are versioned; lab content is fetched
src/content/config.ts      # Zod schema for the docs collection
src/styles/custom.css      # Starlight CSS overrides
astro.config.mjs           # Astro + Starlight config; reads the generated sidebar if present
```

### Fetch pipeline (`scripts/fetch-content.mjs`)

1. Reads `labs.config.json`; for each lab, clones the repo (or copies from a local sibling with `--local`)
2. Copies the lab's `notes/` folder to `src/content/docs/<slug>/`
3. For each `.md`/`.mdx` without a `title` in frontmatter: takes the first `# H1` as `title` and removes it from the body (Starlight renders the title). Falls back to a humanized filename
4. Builds the sidebar: if the content repo has a root `sidebar.json`, its `sections` become groups/links prefixed with the lab slug; otherwise the whole lab directory is autogenerated. Output goes to `labs-sidebar.generated.json`

Notes that already have full frontmatter pass through untouched, so content repos may still use frontmatter for extra fields (`description`, `tags`, `date`).

### Deployment

`.github/workflows/deploy.yml` runs the fetch script, builds and deploys `dist/` to GitHub Pages on: push to `main`, manual dispatch, and `repository_dispatch` with event type `content-update` (sent by content repos when notes change; they need a `HUB_DISPATCH_TOKEN` secret with write access to this repo).

The `base` in `astro.config.mjs` must stay `/labs` to match the Pages sub-path.

## Content conventions (for the content repos)

- Notes are plain Markdown in `notes/`, written in **Portuguese (pt-BR)**
- First line of each note must be the title as `# H1`
- Numeric filename prefixes control sidebar order inside a folder: `01-`, `02-`, ...
- Images live next to the notes (e.g. `notes/secao/assets/img.png`) and are referenced with relative paths in Markdown syntax (`![alt](./assets/img.png)`)
- Internal links between notes must use the full site path (e.g. `/labs/ai-labs/fundamentos/01-introducao-ia/`)
- Optional `sidebar.json` at the content repo root declares the lab's sidebar sections (see README)

## Adding a new lab

1. Content repo: create `notes/` (+ optional `sidebar.json` and the `notify-hub.yml` workflow)
2. Hub: add an entry to `labs.config.json`
3. Optional: add a `LinkCard` for the lab in `src/content/docs/index.mdx`

## Conventions

- Use hyphens (-) instead of em-dashes (—) in all text
- In Markdown, do NOT use `---` as a section separator (except for notes/attributions at the end of a file)
- **Git:** never `git commit` or `git push` automatically - only when explicitly requested
